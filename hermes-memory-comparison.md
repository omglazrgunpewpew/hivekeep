# Hermes vs Hivekeep: A Memory-System Comparison

**TL;DR — the single biggest architectural difference:** Hermes treats long-term memory as a pair of small, human-readable Markdown files (`MEMORY.md` + `USER.md`) that the LLM curates by hand and that are injected verbatim into every system prompt, with retrieval over past conversations delegated to a lexical FTS5 search tool and richer recall pushed out to optional single-select external provider plugins. Hivekeep treats memory as a structured SQLite store with per-row embeddings, where capture is largely automatic (an LLM extraction pipeline runs on every compaction) and recall is a hybrid sqlite-vec KNN + FTS5 pipeline fused by Reciprocal Rank Fusion and re-ranked by decay/importance/recency boosts. In one sentence: **Hermes = curated Markdown + LLM self-curation + pluggable providers + a skills/learning loop; Hivekeep = SQLite + embeddings + automatic extraction + hybrid semantic search.**

## Hermes memory — how it works

### Built-in model & curation

The entire built-in store is two `§`-delimited Markdown files under `~/.hermes/memories/`: `MEMORY.md` (the agent's own environment/convention/lesson notes) and `USER.md` (who the user is, preferences, communication style). There are **no per-fact files and no daily-log files** — just these two flat lists. `MemoryStore` in `tools/memory_tool.py` owns read/write, splitting/joining entries by `ENTRY_DELIMITER = "\n§\n"`. Bounds are **character counts, not tokens** (deliberately model-independent): `memory_char_limit` default 2200, `user_char_limit` default 1375, read in `agent/agent_init.py` (~line 1078). The whole store is budgeted against the limit, so no single entry can exceed it.

Curation is **LLM-driven, not algorithmic**: there is no decay, scoring, eviction, or summarization of entries. When `add` would exceed the limit it returns `success: false` with a "Replace or remove existing entries first" message plus current usage, and the model is expected to consolidate by hand. The large `agent/curator.py` is a red herring for memory — it curates agent-created **skills** (pin/archive/consolidate) on a 7-day inactivity cadence and never touches `MEMORY.md`/`USER.md`.

### Write path & nudges

The `memory` tool (`MEMORY_SCHEMA`) exposes exactly three actions: `add`, `replace`, `remove` (the docs mention a `read` action that does not exist in the schema). `target` ∈ `{memory, user}`. `replace`/`remove` identify an entry by a **unique substring** (`old_text`); ambiguous matches error out asking to be more specific. Writes are hardened: per-file `.lock` (fcntl/msvcrt), `_reload_target()` re-read under lock (sister-session-safe), atomic temp-file + `fsync` + rename, and `_detect_external_drift()` which backs up to `.bak.<ts>` and **refuses** the write if the on-disk file wouldn't round-trip through the `§`-parser or has an over-limit entry (issue #26045). Every write is threat-scanned (`tools/threat_patterns.first_threat_message`, strict scope).

Capture is nudged on a cadence rather than extracted automatically. In `agent/conversation_loop.py` each user turn increments `_turns_since_memory`; at `_memory_nudge_interval` (default 10) it sets `_should_review_memory`. After the response is delivered, `_spawn_background_review` forks a daemon-thread `AIAgent` (`agent/background_review.py`) that shares the parent's `_memory_store`, is whitelisted to memory+skills tools, and is fed `_MEMORY_REVIEW_PROMPT`. Even this "automatic" path resolves to **explicit memory-tool calls**; a "Nothing to save." pass writes nothing. Counter hydration uses modulo (`prior_user_turns % interval`) so the gateway's fresh-agent-per-message model still fires the cadence (issue #22357).

### Recall & context injection

Built-in memory is **always-loaded, non-selective**. At session start `load_from_disk()` reads both files, dedupes (`dict.fromkeys`), threat-sanitizes (poisoned entries become `[BLOCKED: …]` placeholders in the snapshot only), and freezes `_system_prompt_snapshot`. `format_for_system_prompt()` returns this **frozen snapshot**, placed in the "volatile" tier of the three-tier system prompt (`agent/system_prompt.py`, after identity/tool guidance and context files, before the date-only timestamp). The header shows live usage, e.g. `MEMORY (your personal notes) [67% — 1,474/2,200 chars]`. The freeze is a deliberate prefix-cache optimization: mid-session writes hit disk immediately but do **not** change the prompt until the next session start or a post-compression `invalidate_system_prompt()`.

Past conversations are **not** auto-injected. The model recalls them on demand via `session_search` (`tools/session_search_tool.py`), a zero-LLM FTS5/BM25 search over the SQLite `messages_fts` virtual table (`hermes_state.py`). It supports discovery (with goal→match→resolution "bookends"), scroll, read, and browse, with lineage-aware dedup that excludes the current session. CJK queries route to a trigram FTS5 table. If FTS5 is unavailable, search degrades to browse/read only.

### Compaction

`ContextCompressor` (`agent/context_compressor.py`) triggers when estimated prompt tokens cross `threshold_percent` (default 0.50) of the context window, floored at 64K, with anti-thrashing (back off after two passes saving <10%). The algorithm: (1) cheap no-LLM tool-output pruning (dedup, summarize, strip base64, shrink tool-call args inside valid JSON); (2) protect head (`protect_first_n`, default 3) + token-budget tail (never splitting tool-call/result pairs, always keeping the latest user message); (3) an auxiliary "compression" model writes a **structured Markdown checkpoint** (Active Task, Goal, Constraints, Completed Actions, …), using an iterative-update prompt when a prior summary exists; (4) reassemble with a `SUMMARY_PREFIX` "reference-only, latest message wins" handoff. The live middle is lossily replaced, but the SQLite session **splits** (old session ended with reason `compression`, new child session created), so raw transcripts persist. Crucially, just before rotation `commit_memory_session()` fires `on_session_end()` — **compaction directly feeds long-term memory extraction** (best-effort, swallowed on failure).

### External providers

The `MemoryProvider` ABC (`agent/memory_provider.py`) defines a single-select plugin layer that runs **alongside** (never replaces) the built-in store. `MemoryManager` (`agent/memory_manager.py`) enforces "at most one external provider," fans out 12 lifecycle hooks (`prefetch`, `sync_turn`, `on_pre_compress`, `on_delegation`, `on_session_switch`, `on_memory_write`, …) each wrapped in try/except for failure isolation, and mirrors built-in writes out via `on_memory_write`. Recalled context is computed once per turn, wrapped in a `<memory-context>` fenced block, appended to the current user message **for that one API call only** (never persisted), and stripped from model output by a streaming-safe scrubber. The 9 bundled providers add distinct capabilities: honcho (dialectic cross-session user modeling), mem0 (server-side fact extraction), hindsight (knowledge graph + reflection), holographic (local SQLite + HRR algebra + asymmetric trust scoring), supermemory/retaindb/openviking/byterover (semantic/hybrid cloud or hierarchical stores).

### Learning loop & skills

Hermes converts experience into reusable capability through skills — Markdown directory packages (`SKILL.md` + `references/templates/scripts`) under `~/.hermes/skills/`. The same per-turn background-review fork that writes user memory also patches/creates skills (`skill_manage`, `tools/skill_manager_tool.py`), gated by a `skill_provenance` ContextVar so only **agent-created** skills (`created_by="agent"`) are curation-eligible — user-authored skills are never auto-touched. A periodic Curator (`agent/curator.py`, ~7-day idle cadence) runs a non-LLM state machine (active→stale at 30d→archived at 90d, archive-not-delete) plus an LLM "umbrella-building" consolidation pass. Memory and skill reviews have **independent triggers**: memory fires on user-turn count, skills on tool-iteration count. `trajectory.py`/`insights.py` are orthogonal (training dumps / analytics), not part of the live loop.

## Hivekeep memory — how it works

### Storage + dual-channel

Hivekeep stores per-Kin memory in one SQLite `memories` table (`schema.md` L225–254): `content`, an `embedding` blob (`Float32Array` bytes), `category` (`fact|preference|decision|knowledge`), `subject`, `source_channel` (`automatic|explicit`), `importance` (1–10, null≈5), `scope` (`private|shared`), `retrieval_count`/`last_retrieved_at`, and consolidation columns. It is kept in sync with two virtual tables: `memories_fts` (FTS5, **trigger-synced**) and `memories_vec` (sqlite-vec `vec0`, `float[1536]`, **manually** synced inside `createMemory/updateMemory/deleteMemory` in `memory.ts` — a desync risk for any future direct DB write). Embeddings come from the first valid `embedding`-capable provider (`embeddings.ts`); every embedding write is wrapped in try/catch, so a missing provider degrades gracefully to an FTS-only row.

Two write channels feed it. **Automatic**: `compacting.ts:extractMemories` and `consolidation.ts` create `source_channel='automatic'` rows after a vector-dedup check (`isDuplicateMemory`: KNN k=3, cosine distance < 0.15). **Explicit**: the `memorize` tool (`memory-tools.ts`) creates `source_channel='explicit'` rows with model-supplied category/subject/importance/scope. Companion tools: `recall`, `update_memory`, `forget`, `list_memories`, `review_memories`.

### Hybrid KNN/FTS5 search

`searchMemories` (`memory.ts` L569–669) optionally expands the query (multi-query variations grounded in known subjects + a HyDE hypothetical document), then for each query runs `searchByVector` (sqlite-vec KNN, distance ≤ `1 - similarityThreshold` where threshold 0.5) and `searchByFTS` (prefix-match, terms <3 chars dropped, AND→OR fallback) **in parallel**, scoped to `kin_id` OR `scope='shared'`. Results fuse via **Reciprocal Rank Fusion** (`1/(K + rank)`, K=60; FTS hits weighted by `ftsBoost`=0.5 — the semantic channel is privileged). Fused scores are then multiplied by post-fusion boosts: category-aware **temporal decay** (fact/knowledge durable λ 0.1, preference 0.5, decision ephemeral 2.0, importance-gated, floor 0.7), importance weight, log-capped retrieval-frequency boost, subject/entity boost (1.3), bilingual EN/FR regex category-intent boost (1.25, never boosts `fact`), and recency boost (1.5/1.25/1.1). An optional LLM rerank (`llmScore/10 + hybridScore*0.01` — LLM dominates) and adaptive-K trimming finish the pipeline. `getRelevantMemories` injects the top 10 into prompt Block [5] and fires `trackRetrievals`, which feeds the retrieval-frequency boost and importance recalibration.

### Compaction + extraction pipeline

`compacting.ts` keeps each Kin bounded with a token-aware keep-window + telescoping multi-summary compactor. `shouldCompact` compares effective tokens against `min(thresholdPercent% × ctx, triggerMaxTokens)` (defaults 75% / 300k), preferring provider-reported `apiContextTokens` ground truth and **counting `toolCalls` JSON** (a fix for tool-heavy Kins that silently never compacted). `runCompacting` walks backward from newest until a keep-budget (`min(25% × ctx, 100k)`) is exceeded, summarizes everything older into a structured `compacting_summaries` row (`depth=0`, `isInContext=true`), and **never deletes originals**. `maybeMergeSummaries` telescopes the oldest half into a `depth+1` row when count/token budgets are exceeded (originals archived, not deleted). Active summaries inject into system-prompt block 6.9.

Each depth-0 compaction awaits `extractMemories`: it loads **all** existing memories as a dedup index (an unbounded-input risk), prompts an extraction model for a JSON array of `{action: add|update, content, category, subject, importance, updateIndex}`, then routes `update`→`updateMemory` (re-embeds) or `add`→`addIfNotDuplicate`. It is followed by `consolidateMemories` (asymmetric thresholds: same-subject at config threshold, cross-subject at ≥0.95), `recalibrateImportance` (deliberately de-tuned to kill a runaway-popularity feedback loop), and conservative `pruneStaleMemories`.

### Vault redaction blocking

`getNonCompactedMessages` requires `redactPending=false`, structurally excluding any message flagged for pending redaction from the summarize batch so a secret cannot be frozen into a summary before redaction lands. `get_secret` returns a value to the model but never persists it. **Caveat from the findings:** in the current tree no code path actually sets `redactPending=true` — the guard is wired but its deferred-redaction setter is absent; `redact_message` runs synchronously and always writes `redactPending=false`.

### User model (contacts) — Hivekeep's `USER.md` equivalent

Separate from the ranked `memories` table, Hivekeep models *people* in a shared `contacts` registry with per-contact `contact_notes` scoped `global` (all Kins), `private` (one Kin), or `user` (written by the platform user, read-only). At onboarding, `onboarding.ts` auto-creates a contact linked (`linkedUserId`) to each user — including invited users. On every user turn, `kin-engine.ts` (`enrichSpeakerFromContact`) loads the linked contact's notes and `prompt-builder.ts` injects a stable `## Current speaker` block (`[6.75]`, volatile tier) with identity + the three note scopes. When the contact has no global notes, the block escalates to a PRIORITY instruction to get to know the user and persist what's learned via `set_contact_note(contactId, "global", …)` "so all Kins benefit" — i.e. the cross-Kin "don't re-learn the user" guarantee is built in. This is a genuine, always-injected user-model block (richer than Hermes's single-user `USER.md`), and it is the right home for user identity/preferences — the gap vs Hermes is only that it is **not size-bounded**.

## Key differences

The two systems sit at opposite ends of a curation spectrum. **Hermes is opinionated and minimal.** Its built-in memory is a tiny, auditable, char-bounded Markdown store that a human can read and edit, with capture and pruning entirely delegated to LLM judgment (the model, or a forked review agent, decides what to add/replace/remove). The payoff is simplicity, transparency, strong integrity (atomic writes, locks, drift refusal, dual threat scans), and a frozen-snapshot design that keeps the prefix cache hot. The cost is **tiny capacity** (~8–15 + 5–10 entries), **no relevance ranking** (the entire store is injected every turn), no semantic recall in the built-in path, and dependence on the model to consolidate when full — if it doesn't, useful facts are simply dropped. Hermes compensates with two adjacent mechanisms: `session_search` for verbatim transcript recall, and pluggable external providers for anyone who wants a real scalable backend.

**Hivekeep is automatic and quantitative.** Its SQLite + embeddings substrate scales to arbitrarily many facts, captures them automatically during compaction (no reliance on the model remembering to call a tool), and retrieves them by genuine hybrid relevance with a rich, tunable ranking stack. The payoff is recall quality and hands-off operation. The cost is complexity and opacity: a multi-LLM-call retrieval path (multi-query, HyDE, rerank) whose latency/cost scales up, a manually-synced vector table, a hard-fixed 1536-dimension constraint, magic-constant heuristics, and feedback loops (retrieval boost ↔ importance recalibration) the code actively fights. Extraction quality hinges on one JSON-returning LLM call parsed by a greedy regex.

The philosophical divergence is sharpest in two places (a third — user modeling — turns out to be a *convergence*). **User modeling (both have it):** Hermes uses an explicit, always-present `USER.md`; Hivekeep has the equivalent in its `Current speaker` block (`prompt-builder.ts [6.75]`) backed by `contacts` + `contact_notes` — per-speaker identity plus global/user-authored/kin-private notes, injected every turn, with an inline instruction to persist **global** notes "so all Kins benefit" (auto-linked to each user at onboarding). Hivekeep's is in fact *richer* (multi-speaker, scoped, shared across Kins); the only thing Hermes does better here is **hard-bounding** the block so prompt cost stays fixed. **Learning:** Hermes captures *procedural* knowledge (skills) and curates it into reusable umbrellas; Hivekeep captures only *declarative* facts. **Extensibility:** Hermes lets you swap in a knowledge-graph or cloud-semantic backend via one config line; Hivekeep's backend is fixed (only the embedding model is pluggable). Both are defensible — Hermes optimizes for auditability, prefix-cache economics, and a "grows with you" capability loop; Hivekeep optimizes for scale, automatic capture, and semantic recall quality.

## What Hivekeep could borrow

Prioritized, grounded in Hivekeep's real architecture (SQLite, sqlite-vec, FTS5, Bun/Hono, dual-channel).

1. **Bound & curate the existing user-model block — don't build a new one (S).** *Correction:* Hivekeep **already has** Hermes's `USER.md` equivalent, and richer: the `Current speaker` block (`prompt-builder.ts [6.75]`) renders the linked contact's identity + global/user/kin-private notes every turn, a contact is auto-created and linked to each user at onboarding (`onboarding.ts`), and the block already nudges the Kin to save **global** notes "so all Kins benefit". So there is nothing to build here. *The one real gap vs Hermes:* it is **not bounded** — the block does `notes.map(...).join('\n')` with no cap (`prompt-builder.ts [6.75]`) and `enrichSpeakerFromContact` (`kin-engine.ts`) selects all notes for the contact, so a long-lived chatty contact's notes grow unbounded and inflate every prompt. *How:* add a char/token budget when assembling the speaker block (keep most-recent / highest-signal notes), and optionally a Hermes-curator-style periodic pass over `contact_notes` that merges/prunes stale notes (gated by `scope` so user-authored notes are never auto-edited). Small, high-value. *Also (S, prompt-only):* make `CONFIGURATOR_MISSION` explicit that onboarding facts go to `set_contact_note(..., "global", ...)` — today only the speaker block carries the "global so all Kins benefit" emphasis; the configurator's own mission just says "set_contact_note" without the scope.

2. **A curator/decay maintenance pass with provenance (M).** *What:* a periodic, partly non-LLM hygiene job that ages/archives stale memories and merges near-dupes, separating user-authored (`explicit`) from auto-extracted (`automatic`) so explicit memories are protected. *Why:* Hivekeep already has recalibration + pruning but they're fixed-constant and fight feedback loops; a Hermes-style archive-not-delete state machine plus provenance gating would make hygiene safer and auditable. *How:* extend `memory.ts` (`recalibrateImportance`/`pruneStaleMemories`) into a scheduled pass via croner, add an `archived`/`isInContext`-style soft-delete column to `memories`, and gate destructive transitions on `source_channel`. Leverage the existing consolidation thresholds.

3. **Conversation-search recall as a tool (S–M).** *What:* a `session_search`-equivalent FTS5 tool over the `messages` table so the Kin can recall verbatim past exchanges on demand, not only extracted facts. *Why:* extraction is lossy; some recall needs the original wording (commit SHAs, exact phrasings, "what did we decide on X"). Hivekeep already has FTS5 infrastructure and a tool framework. *How:* add a `messages_fts` virtual table + trigger (mirroring `memories_fts`) and a read-only/concurrency-safe `search_conversation` tool in `src/server/tools/`, with bookend/window assembly like Hermes' `get_anchored_view`. Low risk, additive.

4. **Pluggable external memory providers (L).** *What:* a `MemoryProvider`-style interface so power users can attach a knowledge-graph or cloud-semantic backend alongside Hivekeep's built-in store. *Why:* Hivekeep's backend is fixed; an additive plugin layer would let it match Hermes' extensibility without giving up its canonical SQLite store. *How:* define an interface in `src/server/services/`, fan out `sync_turn`/`prefetch`/`on_compact` hooks from `compacting.ts` and the kin-engine, mirror writes via an `on_memory_write` bridge, and fence recalled context at prompt-build time. Large because of the failure-isolation, signature-introspection, and security-fencing surface Hermes proves you need.

5. **Skills-as-memory / procedural capture (L).** *What:* capture reusable "how to do X for this user" procedures (and user style corrections like "stop being verbose") as durable, discoverable artifacts, distinct from declarative facts. *Why:* Hivekeep only stores facts; user frustration and recurring workflows currently can't harden into capability. *How:* a new `skills` table (or Markdown packages under `data/workspaces/<kin>/skills/`), a `skill_manage` tool, a discovery surface in `prompt-builder.ts`, and a consolidation pass. Pair with Hivekeep's strength — index skills with sqlite-vec embeddings for **semantic** discovery, which Hermes notably lacks. Large and arguably a product decision, not just an engineering one.

6. **Harden the vector-table sync; treat the redaction guard as forward-looking scaffolding (S).** *What:* make `memories_vec` trigger-synced (or add an integrity check), and decide what to do with the dormant `redactPending` guard. *Why (verified against the tree):* `memories_vec` is synced by explicit `INSERT`/`DELETE` inside `createMemory`/`updateMemory`/`deleteMemory`/`reembedAllMemories` (no SQL trigger, unlike trigger-synced `memories_fts`), so any *future* direct DB write would silently desync the vector index — a latent risk, not a current bug since all writes go through those functions today. Separately, `redactPending` is only ever written `false` (schema default + `vault.ts`, `compacting.ts`, `kins.ts`) and `redactMessage` redacts **synchronously**, so the `compacting.ts:265` `redactPending=false` filter is **dormant scaffolding** — it is *not* a live security hole (there is no window where a secret is pending-but-unredacted during compaction), but it only becomes meaningful if deferred/async redaction is ever introduced. *How:* either move vec sync into a trigger or add a consistency assertion in `reembedAllMemories`; and either delete the unused `redactPending` plumbing or, if async redaction is on the roadmap, add the setter that flips it `true`. Small, high-confidence cleanups.
