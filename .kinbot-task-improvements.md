# KinBot sub-Kin task system — improvement loop

Durable scratchpad for the /loop session that iteratively improves task agents. Each iteration appends to **Shipped**, updates **Queued**, and trims **Pending** as items are picked up.

## Baseline (2026-05-15, before loop)

Real prod task #32 (`feat: git-status snapshot injection`, ticket 32) executed by the sub-Kin agent immediately after the first batch of prompt improvements shipped (commit `45d756ba`). Tool-call waste observed:

- **~25 calls** of PATH archaeology (`which bun`, `ls ~/.bun/bin`, then 20+ `export PATH=$HOME/.bun/bin:$PATH && ...` prefixes). Most visible regression.
- **~15 redundant greps** (variations on `projects`, `workspacePath`, `ticketAssignment`, `githubUrl`).
- **~10 redundant file re-reads** (tasks.ts × 4, prompt-builder.ts × 3, projects.ts × 2, prompt-builder.test.ts × 2).
- **~12 bash-wrapper violations** despite the new prompt rules (`cat migration.sql`, `head -20 kins.ts`, `ls | tail`, `grep -n ... in file`).
- User reported `report_to_parent` and `request_input` do nothing visible for ticket sub-Kins.
- Wins from `45d756ba`: agent paused on pre-commit failure instead of `--no-verify` bypass; no silent push to main. **Safety improved, efficiency barely moved.**

### Rejected / intentionally skipped

- **TicketCard "X/Y todos" badge** — kept on queue. Aggregating per-task todos across a ticket's tasks needs more thought (multiple tasks per ticket, in-memory tracker vs DB), and the task panel already surfaces the plan once the user clicks in. Low ROI for a non-trivial change.
- **Ticket / project tool gap-fill** — no real gap surfaced during the loop. The `code` preset already exposes the right ticket/project tools.
- **MORE prompt trimming** — at 6.4 KB the sub-Kin prompt is acceptable. Further cuts risk losing signal.
- **Docs (CLAUDE.md update with the new sub-Kin discipline)** — useful for future contributors but doesn't move the needle on the agent's behaviour. Skipped.

### Honest residual gaps

- The agent CAN still ignore prompt guidance even with all the guards. Hard refusals (bash wrappers, banned commands, read-before-edit) catch the worst patterns; soft hints (duplicate reads/greps, prompt rules) rely on model attention.
- `task_todos` adoption is up to the model. There's a prompt nudge but no hard requirement. On simple tasks the model may skip it (correct); on complex tasks it may forget (wrong).
- The duplicate-call hint doesn't reach across compaction boundaries: a compacted summary might mention "I read foo.ts" but the tracker still considers a fresh read non-duplicate. Bug risk: low (since post-compact reads are typically genuine refreshes).
- The telemetry counters are server-log only, no UI. Next polish iteration would surface them on TaskPanelContent.

4. If the numbers are good, declare (c) met. If not, share the new task's tool-call log and we iterate on the actual failure mode, not the imagined one.
